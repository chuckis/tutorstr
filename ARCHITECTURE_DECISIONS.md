## ADR-001: Разделение слоёв уведомлений и данных

### Контекст

При добавлении системы уведомлений курсор «последнего визита» (`notificationCursor`)
был случайно использован как `since`-фильтр в подписках репозиториев данных.
Это привело к тому, что все события старше последнего визита пользователя
перестали загружаться.

### Решение

Курсор уведомлений (`getNotificationSince`) используется **только** внутри
логики уведомлений. Репозитории данных не знают о его существовании.

### Правило

```
Данные → репозиторий → eventBus → хук данных
Уведомления → NotificationContext → фильтр по since → toast
```

Эти два потока никогда не пересекаются на уровне фильтрации.

## ADR-002: Публичные реле ненадёжны для addressable events

### Контекст

`relay.damus.io` и `relay.primal.net` молча игнорировали широкие подписки
вида `{ kinds: [30001] }` без `authors`. События не приходили, ошибок не было.
Проблема проявлялась только при загрузке — новые события после публикации
приходили нормально.

### Решение

Для критичных addressable events (kinds 30000–30006) добавлен явный метод
`fetchAll()` в репозитории, который выполняет прямой REQ при монтировании
компонента, не полагаясь только на глобальную подписку.

### Правило

Широкий фильтр `{ kinds: [N] }` без `authors` на публичных реле — ненадёжен.
Для исторических данных: явный запрос при монтировании + глобальная подписка
для live-обновлений.

Долгосрочно: своё реле (strfry/nostream) для гарантированного хранения
кастомных kinds.

## ADR-003: Тег `["p"]` — это адресат, не автор

### Контекст

При отмене урока тутором в `publishBookingStatus` передавался `tutorPubkey`
вместо `studentPubkey`. В результате тег `["p"]` содержал pubkey тутора,
и студент не получал событие — фильтр `isMentioned` не срабатывал.

### Решение

```typescript
// было (баг)
const recipient = status === "cancelled" ? request.tutorPubkey : request.pubkey;

// стало
const recipient = request.pubkey; // всегда studentPubkey
```

### Правило

Тег `["p"]` в Nostr означает **адресат** события. Автор и так известен из
`event.pubkey`. При публикации статуса всегда задавай вопрос:
«Кому предназначено это событие?» — и именно этот pubkey идёт в `["p"]`.

## ADR-004: eventBus должен поддерживать replay

### Контекст

Первоначальный eventBus был чистым pub/sub без буфера. При быстром подключении
к реле события приходили до монтирования компонентов и терялись.

### Решение

Zustand-store как буфер событий. `addKindListener` при регистрации нового
слушателя немедленно воспроизводит все уже накопленные события из store.

```typescript
export function addKindListener(kind, listener) {
  // регистрация...
  
  // replay накопленных событий
  const events = useEventBusStore.getState().eventsByKind[kind];
  if (events) {
    for (const event of Object.values(events)) {
      listener(event);
    }
  }
}
```

### Правило

pub/sub без replay — антипаттерн для асинхронных источников данных.
Новый подписчик должен получить уже накопленные события независимо от
момента подписки.

## ADR-005: `limit` в фильтре реле ≠ гарантия получения всех событий

### Контекст

`limit: 1000` в глобальной подписке интерпретировался как «последние 1000
событий по времени». При высокой нагрузке на реле события 3-дневной давности
не попадали в это окно.

### Решение

Заменить `limit` на `since` с явным временным горизонтом:

```typescript
{ kinds: ALL_KINDS, since: Math.floor(Date.now() / 1000) - 30 * 24 * 3600 }
```

### Правило

`limit` в Nostr — подсказка реле, не гарантия. Реле вправе вернуть меньше
событий или проигнорировать параметр. Для исторических данных используй
`since` с явным горизонтом.

## ADR-006: addEventListener требует сохранения ссылки на handler

### Контекст

В `useNotificationCursor` анонимная стрелка передавалась в `addEventListener`,
а именованная функция — в `removeEventListener`. Разные ссылки → слушатель
никогда не удалялся → утечка памяти при каждом ремаунте.

### Решение

```typescript
// было (утечка)
document.addEventListener('visibilitychange', () => {
  if (document.hidden) save();
});
// ...
document.removeEventListener('visibilitychange', save); // другая ссылка!

// стало
const onVisibilityChange = () => {
  if (document.hidden) save();
};
document.addEventListener('visibilitychange', onVisibilityChange);
// ...
document.removeEventListener('visibilitychange', onVisibilityChange); // та же ссылка
```

### Правило

Всегда сохраняй handler в переменную до передачи в `addEventListener`.
Одна переменная — один `add`, один `remove`.

## ADR-007: nostr-tools v2 API

### Контекст

В nostr-tools v2 API `SimplePool` изменился по сравнению с v1.

### Актуальное состояние (v2.19.4)

- `pool.subscribeMany(relays, filter, params)` — принимает **один** `Filter`, не массив
- `pool.subscribe(relays, filter, params)` — аналогично
- Динамические тег-ключи (`#e`, `#p`) имеют тип `string[]`, не `number[]`

```typescript
// правильный тип для совместимости
import type { Filter } from 'nostr-tools';
export type NostrFilter = Filter;
```

### Правило

При обновлении nostr-tools всегда проверяй типы `pool.*` методов —
API менялось между мажорными версиями.

## Общие принципы работы с Nostr

1. **Реле — ненадёжный транспорт.** Могут молчать, терять события,
   игнорировать фильтры без ошибок. Архитектура должна это учитывать.

2. **Явные запросы при монтировании** критичных компонентов — не полагайся
   только на глобальную подписку.

3. **Локальный кэш** (localStorage) как fallback пока реле не ответило.
   Показывай закэшированные данные сразу, обновляй когда придут свежие.

4. **Идемпотентный приём** — одно и то же событие можно получить дважды,
   это норма. Дедупликация по `event.id` обязательна.

5. **Разделение слоёв** — транспорт, данные, уведомления никогда
   не смешиваются в одном слое.

6. **Своё реле** — единственная гарантия хранения кастомных kinds долгосрочно.
